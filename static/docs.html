Getting Started

ChekReg is a privacy-first, locally-executed digital footprint auditor. It was built by taraladka as a personal tool to map an email-based digital footprint without needing to share inbox access with any external service. It worked well enough that it seemed worth sharing openly with anyone who finds it useful.

OSINT vs. CSINT: While ChekReg employs Open-Source Intelligence workflows — such as cross-referencing public breach databases — its primary data source is an authenticated, private email inbox. As that seed data is not publicly accessible, the methodology falls under Closed-Source Intelligence (CSINT).

If you are viewing this page from a running local dashboard, the steps below have already been completed. They are retained here for reference and for users accessing this documentation externally.

1. Prerequisites

  • Python 3.8 or later
  • IMAP access enabled on the target email account

2. Installation

  1. Clone the repository: 
    git clone https://github.com/taraladka/ChekReg.git
cd ChekReg-IMAP
  2. Create and activate a virtual environment: 
    # Windows
python -m venv venv
venv\Scripts\activate

# macOS / Linux
python3 -m venv venv
source venv/bin/activate
  3. Install dependencies: 
    pip install --upgrade tldextract colorama flask duckduckgo-search requests

3. Generating an App Password

Modern email providers block third-party clients from authenticating with a primary account password. ChekReg requires an application-specific password, which grants scoped IMAP access without exposing your primary credentials.

Gmail

  1. Navigate to your Google Account Security page.
  2. Confirm that 2-Step Verification is active.
  3. Search for "App passwords" using the account search bar.
  4. Generate a new password and label it ChekReg.
  5. Google will issue a 16-character token (e.g., abcd efgh ijkl mnop). Store it securely before closing the dialog.

Microsoft Outlook / Hotmail

  1. Navigate to Microsoft Account Advanced Security Options.
  2. Ensure Two-step verification is enabled.
  3. Under "App passwords", select Create a new app password.
  4. Copy the generated token and store it securely.

Yahoo Mail & AOL

  1. Navigate to the Yahoo Account Security page.
  2. Verify that 2-Step Verification is enabled under "How you sign in".
  3. Select Generate and manage app passwords.
  4. Enter ChekReg as the application name and generate the password.
  5. Copy the 16-character token (spaces omitted) into ChekReg when prompted.

Apple iCloud Mail

  1. Sign in to your Apple ID account.
  2. Under Sign-In and Security, select App-Specific Passwords.
  3. Click the + button to generate a new password.
  4. Label it ChekReg and confirm with your Apple ID credentials if prompted.
  5. Copy the generated 16-character string before dismissing the dialog.

Custom or Private IMAP Providers

If you operate a self-hosted or corporate IMAP server, app passwords may not be required depending on server configuration. Use your standard credentials, or contact your system administrator if authentication fails.

4. Running ChekReg

Launch the engine from the project root:

python chekreg.py

The CLI will guide you through the following steps:

  1. Interface selection: Enter 2 to launch the local Flask web interface on localhost:5000, or 1 for terminal output.
  2. Authentication: Provide the target email address and the generated App Password.
  3. IMAP endpoint: Select your email provider from the list.

The scan executes entirely on your local machine. Credentials are stored in volatile memory and are irrecoverably destroyed upon process termination.

Privacy & Security Model

Many tools that help you manage your digital footprint require connecting your inbox to an external service via OAuth. That is a perfectly reasonable approach for many people, and those services offer real value. ChekReg simply takes a different path — one that keeps everything local. No data leaves your machine during a scan.

ChekReg operates under a Zero-Trust, Local-First architecture. Your credentials and email metadata stay entirely on your own hardware.

1. Local Execution

ChekReg runs as a Python process on your own hardware. There are no intermediary servers.

  • No External Routing: Email data is never transmitted to or processed by any server other than your provider's IMAP endpoint.
  • No Telemetry: The engine collects no usage data, crash reports, or analytics of any kind.
  • Direct IMAP Connection: Authentication is performed directly from your machine to your provider's IMAP server over an encrypted SSL/TLS tunnel on port 993.

2. Ephemeral Credential Handling

ChekReg is designed to hold credentials for the minimum duration necessary to complete a scan.

  • Volatile Storage: The App Password is held exclusively in process RAM. It is never written to disk, a configuration file, a database, or any persistent cache.
  • Process Termination: Issuing Ctrl+C or closing the terminal window immediately destroys the process and all in-memory credentials. Additionally, the New Scan button in the web interface and the Back to Scanner button on this page both execute an explicit browser cache flush, clearing all sessionStorage data before returning to the input screen. No residual scan data is retained in the browser.

3. Minimal Privilege Data Extraction

ChekReg requests only the subset of mailbox data required for its analysis.

  • Header-Only: The engine issues RFC822.HEADER fetch commands. Email bodies and attachments are never requested or read. Only the From:, List-Unsubscribe:, and Date: fields are parsed.
  • Read-Only Mode: The IMAP session is opened with readonly=True. The engine is structurally incapable of modifying, flagging, or deleting any message.

4. Have I Been Pwned Integration

Breach checking is entirely opt-in. If enabled, the engine makes outbound HTTPS requests to the haveibeenpwned.com API.

  • Only the domain names extracted from your footprint are transmitted — never your email address or any message content.
  • If checking email-level breach exposure, the request is made to the official HIBP /breachedaccount endpoint over HTTPS.
  • Disabling the HIBP toggle results in a fully offline scan, resolved exclusively against the local data/sites.json database.

Labels & Metrics

ChekReg assigns each discovered service a set of descriptive labels based on what was found in the mailbox. Because labels represent independent properties, a single service can carry multiple labels simultaneously. As a result, the sidebar counts reflect overlapping sets and will not sum to the total service count.

1. Type Labels

Each service receives exactly one type label, determined by the links present in its associated emails:

  • General Account — A direct deletion URL is available in the sites.json database, but no List-Unsubscribe header was detected.
  • Newsletter — A List-Unsubscribe link was found, but no deletion URL is recorded in the database.
  • Hybrid — Both an unsubscribe mechanism and a direct deletion URL are available. This is the most actionable classification.
  • Other — Neither a deletion URL nor an unsubscribe link could be determined.

2. State Labels

State labels are applied independently of type and may co-exist with any combination of other labels:

  • Ghost — No email activity has been detected from this service in the past six months. Dormant accounts represent an elevated risk of unauthorised access.
  • Compromised — The service domain appears in a known public data breach, as reported by Have I Been Pwned.
  • Spam & Clutter — A notably high volume of emails has been received from this sender. This could be newsletters, promotions, or automated alerts — worth reviewing if your inbox feels noisy.

Illustrative Example

Consider a news digest service you subscribed to several years ago. If that service has experienced a data breach, and no email has been received from them in over six months, it would simultaneously carry the Newsletter, Ghost, and Compromised labels — three independent properties applied to a single entity.

System Architecture

ChekReg is structured as a three-layer system: a core analysis engine, a lightweight local web server, and a dependency-free frontend client. All processing occurs on the user's machine.

Data Flow Overview

  1. IMAP Connection: Establishes an authenticated, read-only IMAP4_SSL session directly with the provider's mail server.
  2. Header Extraction: Fetches RFC822.HEADER payloads exclusively; email bodies are never requested.
  3. Domain Resolution: Parses sender domains using TLD extraction and cross-references them against data/sites.json.
  4. Heuristic Scoring: Evaluates per-service metadata — last activity date, breach status, volume — to compute a safety profile.
  5. Result Streaming: Serialises the resolved footprint as a JSON payload and streams it to the frontend over a local WebSocket connection.

1. Core Engine (engine.py)

The FootprintEngine class in engine.py orchestrates all analysis phases.

Phase 1 — Authentication

Binds to the provider's IMAP server on port 993 via IMAP4_SSL. Standard password authentication is rejected by most modern providers; an App Password is therefore required for all supported platforms.

Phase 2 — Metadata Extraction

To minimise both latency and data exposure, only email headers are fetched. The engine parses three specific fields:

  • From — Identifies the originating service domain.
  • List-Unsubscribe — Captures opt-out mechanisms for newsletter classification.
  • Date — Provides the timestamp required for ghost account detection.

Phase 3 — Database Resolution

Each extracted domain is looked up against data/sites.json, which maps domains to human-readable service names, categories, and direct account deletion endpoints (delete_url).

2. Web Server (server.py)

An ephemeral Flask instance serves the frontend on localhost:5000. The server is strictly local and has no external network exposure.

Live scan progress is delivered via flask_sock WebSockets. Two frame types are emitted during a scan:

  • status — Incremental progress updates: {"type": "status", "current": 200, "total": 5000}
  • complete — The fully resolved footprint payload, consumed by the frontend for rendering.

3. Frontend Client (static/)

The interface is implemented in plain HTML, CSS, and JavaScript with no runtime dependencies or build steps.

  • Auditable by Design: The absence of third-party frameworks means any user can inspect the full client source in a text editor. There are no minified bundles or opaque dependencies.
  • Dynamic Rendering: script.js consumes the boolean flags in the backend payload to render and filter service cards client-side without additional server round-trips.

Contributing

At this stage, the only area open for external contributions is the service database — data/sites.json. The codebase itself is not open to pull requests at the moment, and that may change in the future.

The database is still quite small. If you come across a service that ChekReg doesn't recognise, adding it is genuinely useful and straightforward — no code required.

Expanding the Database (sites.json)

Each entry in data/sites.json maps a root domain to a human-readable name, a category, and a direct account deletion or Data Subject Access Request (DSAR) URL.

Entry Format

{
    "canva.com": {
        "name": "Canva",
        "category": "design",
        "delete_url": "https://www.canva.com/settings/your-account"
    }
}

How to Add a Service

  1. Identify the root domain the service uses as its sender address (e.g., marketing.uber.com resolves to uber.com via the TLD extractor).
  2. Find the official page where an account holder can submit a deletion request or DSAR. Prefer first-party settings pages.
  3. Add the entry in alphabetical order within the JSON object.
  4. Submit a Pull Request against the main branch with a brief note on what was added.

The Auto-Resolver Engine

ChekReg includes an integrated 4-stage pipeline to automatically discover account deletion endpoints for services not yet cataloged in sites.json. You can trigger this engine directly from the Web Dashboard (via the Auto-Resolve button) or from the CLI Developer Tools menu.

How it Works

  • Stage 1: JustDeleteMe Database — A fast, offline cross-reference against the open-source JustDeleteMe dataset.
  • Stage 2: DNS Validation — Verifies domain resolution to quickly discard offline, parked, or dead domains to save processing time.
  • Stage 3: Well-known Path Probing — Issues fast HEAD requests to common URL patterns (e.g., /settings/account, /account/delete, /unsubscribe).
  • Stage 4: Search Engine Fallback — Utilizes DuckDuckGo to query for account management pages. It employs strict path-filtering heuristics to ensure the grabbed link is an actionable settings or deletion page rather than a generic privacy policy.
Resolver Accuracy (~85%): The Auto-Resolver is highly effective but not perfect. Accuracy hovers around 85%. Search engine fallbacks can occasionally pull outdated URLs, and advanced anti-bot protections on some websites might block the path-probing stage. We highly encourage developers to manually verify the resolved links and submit a PR to improve the shared database!

We Need Your Contributions

The core of ChekReg's capability relies on the community-driven data/sites.json database. The internet is vast, and many obscure regional or niche services are still missing.

If the Auto-Resolver successfully finds links for your missing accounts, please consider contributing them back! Every new verified link helps hundreds of other users cleanly erase their footprint.

How to Contribute

  1. Run the Auto-Resolver from the Web UI or CLI.
  2. Verify the newly appended links in your data/sites.json file.
  3. Fork the repository and submit a Pull Request against the main branch. No code changes required—just the raw JSON updates!

CLI Arguments & Flags

You can bypass the interactive menu by launching ChekReg using the following flags:

  • python chekreg.py - Starts the interactive selector menu.
  • python chekreg.py --web - Immediately launches the Web Dashboard.
  • python chekreg.py --cli - Immediately launches the Terminal Footprint Mapper.
  • python chekreg.py --help - Shows the help output and CLI information.
